## 1. アプリ概要

### 1.1 コンセプト

#### アプリ名(仮): money-log-app

**「実感を残す家計簿」**

口座連携による自動取得を行わず、ユーザ自身が支出を入力することで「今いくら使っているか」の実感を保つことを最優先とする家計簿アプリ。ただし、サブスクのように完全に固定的な支出は自動計上し、光熱費のように毎月発生するが金額が変動する費用は入力を促す仕組みを設けることで、手入力の負担を最小限に抑える。

### 1.2 設計判断の指針

機能追加や設計の選択で迷ったときは、「これは実感を損なわないか?」を判断基準とする。

### 1.3 ターゲット

- 一次的には開発者自身による個人利用
- 将来的な一般公開も視野に入れる

## 3. 機能一覧（MVP）

### 3.1 支出管理

支出は性質に応じて以下の3タイプで扱う。内部データとしてはすべて「支出レコード」として統一されるが、入力経路と計上方法が異なる。

| タイプ | 額 | 日付 | 計上方法 |
|--------|-----|------|----------|
| 都度支出 | 変動 | 不定 | 手動入力 |
| サブスク | 固定 | 固定 | 自動計上 |
| 変動固定費（毎月費用） | 変動 | 固定 | バナー＋通知で入力促し |

#### 3.1.1 都度支出の入力項目

- 日付
- タイトル（内容）
- 金額
- カテゴリ
- 支払い方法
- メモ

#### 3.1.2 サブスクの登録・自動計上

- ユーザは「サブスクマスター」に名前・金額・毎月の計上日・カテゴリ・支払い方法を登録する
- **アプリ起動時**に「今日以前で未計上のサブスクがあるか」をチェックし、該当があれば支出レコードとして自動追加する
- 自動計上済みかどうかは各サブスクごとに「最終計上月」で管理する
- 重複防止のため、支出レコードに「由来キー（`sourceKey`）」を付与する（形式: `sub-{id}-YYYYMM`）
- 自動計上された支出レコードはその後、通常の支出と同じように**編集・削除が可能**
- ユーザが自動計上された支出レコードを削除しても、最終計上月は進んだままのため**再計上されない**
- 月末日の扱い: 計上日として31日が指定されている場合、31日が存在しない月は**その月の最終日に繰り上げ**て計上する

#### 3.1.3 変動固定費（毎月費用）の登録・入力促し

- ユーザは「変動固定費マスター」に名前・毎月の入力促し日・カテゴリ・支払い方法を登録する（金額は登録しない）
- 指定日になったらホーム画面にバナーを表示し、プッシュ通知でも入力を促す
- **入力済みの判定**: 支出レコードの `sourceKey`（形式: `vfc-{id}-YYYYMM`）を検索して判定する。バナー/アラート経由で入力した時のみ `sourceKey` が付与される
- ユーザが金額を入力した時点で通常の支出レコードとして確定する

**バナーとアラートの役割分担:**

| | バナー | アラート帯＋専用画面 |
|---|---|---|
| 対象 | **当月**の未入力 | **過去月**（先月・先々月）の未入力 |
| 表示位置 | ホーム画面上部（最大2件＋残件数） | ホーム最上部のコンパクトな帯 |
| アクション | タップで入力モーダル | 入力 or 無視 |
| 消える条件 | 入力されたら消える | 入力 or 無視されたら消える |
| 追跡期間 | 当月のみ | 過去2ヶ月（当月含め合計3ヶ月） |

- 過去月の「無視」操作は専用エンティティ（`IgnoredVariableFixedCost`）で記録する

### 3.2 収入管理

- 入力項目: 日付・タイトル・金額・カテゴリ・メモ
- カテゴリのプリセット: 給与・ボーナス・その他
- カテゴリはユーザが追加可能
- 収入に支払い方法（入金先）の分類は**持たない**

### 3.3 表示・集計

#### 3.3.1 ホーム画面のサブタブ構成

| サブタブ | 表示内容 |
|---------|----------|
| 日別（デフォルト） | 日別のリスト表示 / カレンダー表示の切り替え |
| カテゴリ別 | 上半分に円グラフ、下半分にカテゴリ別リスト（当月のみ） |
| 月間 | 月の合計支出、1日平均、月ごとの推移グラフ（直近6ヶ月、棒グラフ） |

#### 3.3.2 家計簿一覧画面

ホームとは役割を分け、**検索・フィルタ専用の画面**として機能する（支出のみ表示）。

フィルタ構成:

| 種類 | 表示方式 | 内容 |
|------|---------|------|
| 年月フィルタ | 常時表示 | プリセット（今月/先月/直近3ヶ月/直近6ヶ月/今年/カスタム） |
| カテゴリ | 常時表示 | カテゴリ選択 |
| 詳細フィルター | 折りたたみ | 金額範囲・キーワード・支払い方法 |

- キーワード検索はひらがな/カタカナ同一視
- 期間は半開区間 `[start, end)` で定義（取りこぼし防止）

### 3.4 設定

- カテゴリの追加・編集・削除
- 支払い方法の追加・編集・削除
- サブスクの管理
- 変動固定費（毎月費用）の管理
- 通知設定（ON/OFF、通知時刻）
- データのエクスポート（CSV / JSON）
- テーマカラー
- 週の始まり（日曜/月曜）

### 3.5 補助機能

**簡易電卓**: 支出登録時に金額欄から電卓モーダルを呼び出し、四則演算の結果を金額に反映できる。画面半分の高さのシートとして表示し、最大8桁（約1億円）まで対応。小数は非対応（円単位）。

## 5. データ概念モデル

### 5.1 エンティティ一覧

| エンティティ | 内部名 | 役割 |
|-------------|--------|------|
| 支出・収入レコード | Transaction | 個々の支出・収入（都度入力・サブスク自動計上・毎月費用の確定後を含む） |
| カテゴリ | Category | 支出・収入それぞれのカテゴリ（プリセット＋ユーザ追加） |
| 支払い方法 | PaymentMethod | 現金、PayPay、クレジットカードなど（プリセット＋ユーザ追加） |
| サブスクマスター | Subscription | 定期的に自動計上するサブスクの定義 |
| 変動固定費マスター | VariableFixedCost | 指定日に入力促しを出す毎月費用の定義 |
| 無視記録 | IgnoredVariableFixedCost | 毎月費用の「無視」操作を記録するマーカー |

### 5.2 エンティティ関連

- Transaction → Category（オプショナル）
- Transaction → PaymentMethod（オプショナル）
- Subscription → Category（オプショナル）
- Subscription → PaymentMethod（オプショナル）
- VariableFixedCost → Category（オプショナル）
- VariableFixedCost → PaymentMethod（オプショナル）
- IgnoredVariableFixedCost → VariableFixedCost（UUID で間接参照）
- Subscription / VariableFixedCost → Transaction への直接参照は持たない（`sourceKey` で間接的に紐付く）
- 全リレーションは双方向オプショナル、削除ルールは `.nullify`

### 5.3 プリセット

**支出カテゴリ**（SF Symbols アイコン付き）:

| 名前 | アイコン | 備考 |
|------|---------|------|
| 食費 | fork.knife | |
| 日用品 | basket | |
| 交通費 | tram.fill | |
| 娯楽 | gamecontroller | |
| サブスク | arrow.clockwise | |
| その他 | ellipsis.circle | 削除不可 |

**収入カテゴリ**:

| 名前 | アイコン | 備考 |
|------|---------|------|
| 給与 | yensign.circle | |
| ボーナス | gift | |
| その他 | ellipsis.circle | 削除不可 |

**支払い方法**: 現金 / PayPay / クレジットカード

### 5.4 カテゴリ・支払い方法の削除ルール

**カテゴリの削除:**
- 「その他」カテゴリは削除不可
- それ以外はプリセットも含め削除可能
- 削除時、使用中の Transaction / Subscription / VariableFixedCost のカテゴリ参照を「その他」に付け替え、確認アラートを表示
- 同一種別（支出/収入）内で名前の重複は不可

**支払い方法の削除:**
- 全ての支払い方法が削除可能（プリセットも含む）
- ただし**最後の1件は削除不可**
- 削除時、使用中の Transaction / Subscription / VariableFixedCost の支払い方法参照を `nil`（未設定）に変更
- 一覧表示では `nil` を「未設定」と表示
- 同名の重複は不可

## 6. 主要な業務ルール

### 6.1 月の締め

月初〜月末（1日〜末日）で固定。給料日締めなどのカスタマイズには対応しない。

### 6.2 クレジットカードの計上タイミング

クレジットカードで支払った支出は、**実際に使った日（利用日）に計上する**。引き落とし日への繰り延べは行わない。

**理由**: コンセプトである「実感を残す」に基づき、「いつ何に使ったか」を優先するため。月の支出合計は実際のキャッシュフローとは一時的にズレるが、これは仕様とする。

### 6.3 自動計上・通知の発火タイミング

- サブスク自動計上: アプリ起動時にチェック（CloudKit 同期を考慮し、起動直後と数秒後の2段構え）
- 変動固定費バナー: アプリ起動時にチェックして表示
- 変動固定費のプッシュ通知: 毎月繰り返しのローカル通知（デフォルト 20:00）

### 6.4 由来キー（sourceKey）によるレコード管理

支出レコードには `sourceKey` フィールドを設け、サブスク自動計上と毎月費用入力の由来を管理する。

| 由来 | sourceKey の形式 | 用途 |
|------|-----------------|------|
| サブスク自動計上 | `sub-{subscriptionId}-YYYYMM` | 重複計上の防止 |
| 毎月費用入力 | `vfc-{variableFixedCostId}-YYYYMM` | 入力済み判定 |
| 都度支出 | nil | — |

### 6.5 通知の権限取得

iOS の通知権限ダイアログは1度拒否されると再表示できないため、**2段階パーミッション方式**を採用する。アプリ内の確認アラートで「はい」を選んだ場合のみ、iOS の権限ダイアログを表示する。

## 7. 非機能要件

### 7.1 データ保護

金銭データを扱うため、データの喪失リスクを最小化する。

- **プライマリ保存**: 端末ローカル + iCloud 同期（CloudKit private database）
- **セカンダリバックアップ**: 設定画面から手動で CSV / JSON エクスポートできる機能を用意

### 7.2 オフライン動作

外部APIに依存しないため、すべての機能はオフラインで完結する。

### 7.3 プライバシー

口座連携やサーバ送信を行わないため、金銭データは端末およびユーザのiCloudアカウント内にのみ存在する。

### 7.4 CloudKit 同期時の制約

- 全ての非リレーションフィールドはオプショナルまたはデフォルト値付き
- `@Attribute(.unique)` は使用不可（CloudKit がユニーク制約をサポートしない）
- リレーションはオプショナル必須
- 削除ルールは `.nullify` のみ（`.cascade` 不可）

## 8. スコープ外（将来拡張候補）

以下は基本設計の対象外とするが、将来的な拡張可能性を考慮してデータモデルに拡張ポイントを残している。

### 8.1 AI予算アドバイス機能

月末の予測や気づきの提示など、LLMを活用した予算管理機能。単純な予算設定はコンセプト上の効果が薄いため、AI連携とセットで将来検討する。

### 8.2 グループ共有機能（旅行モード）

- 方式: リンク共有による非リアルタイム編集型
- 用途: 旅行など一時的な複数人での立て替え管理
- 準備: Transaction に `groupId: UUID?` フィールドを用意済み

## 9. 基本設計の合意事項まとめ

- コンセプトは「実感を残す家計簿」。口座連携はしない
- 支出は「都度 / サブスク / 変動固定費（毎月費用）」の3タイプで扱う
- カテゴリは SF Symbols アイコン付きで、プリセット＋ユーザ追加方式
- 支払い方法はプリセット＋ユーザ追加方式
- カテゴリ削除時は「その他」に付け替え、支払い方法削除時は `nil`（未設定）に変更
- ホームは3サブタブ（日別・カテゴリ別・月間）
- 家計簿一覧は検索・フィルタ専用画面として独立（支出のみ）
- クレジットカードは利用日で計上する
- サブスク自動計上と毎月費用入力は `sourceKey` で由来管理・重複防止する
- 毎月費用のバナーは当月分、過去月はアラート帯＋専用画面で対応（追跡範囲: 直近3ヶ月）
- データ保護は iCloud 同期＋手動エクスポートの二重体制
- iPhoneをメインターゲットとし、5タブ形式のナビゲーション＋ハンバーガーメニュー
- 永続化は SwiftData、iOS 18.0 以上
- 通知権限は2段階パーミッション方式
- AI予算機能・グループ機能は将来拡張として今回は実装しない
