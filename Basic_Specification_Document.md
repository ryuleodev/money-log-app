## 1. アプリ概要

### 1.1 コンセプト

#### アプリ名(仮): money-log-app

**「実感を残す家計簿」**

口座連携による自動取得を行わず、ユーザ自身が支出を入力することで「今いくら使っているか」の実感を保つことを最優先とする家計簿アプリ。ただし、サブスクのように完全に固定的な支出は自動計上し、光熱費のように毎月発生するが金額が変動する費用は入力を促す仕組みを設けることで、手入力の負担を最小限に抑える。

### 1.2 設計判断の指針

機能追加や設計の選択で迷ったときは、「これは実感を損なわないか?」を判断基準とする。

### 1.3 ターゲット

- 一次的には開発者自身による個人利用
- 将来的な一般公開も視野に入れる

## 3. 機能一覧(MVP)

### 3.1 支出管理

支出は性質に応じて以下の3タイプで扱う。内部データとしてはすべて「支出レコード」として統一されるが、入力経路と計上方法が異なる。

| タイプ | 額 | 日付 | 計上方法 |
|--------|-----|------|----------|
| 都度支出 | 変動 | 不定 | 手動入力 |
| サブスク | 固定 | 固定 | 自動計上 |
| 変動固定費 | 変動 | 固定 | バナー+通知で入力促し |

#### 3.1.1 都度支出の入力項目

- 日付
- タイトル(詳細)
- 金額
- カテゴリ
- 支払い方法
- メモ

#### 3.1.2 サブスクの登録・自動計上

- ユーザは「サブスクマスター」に名前・金額・毎月の計上日を登録する
- **アプリ起動時**に「今日以前で未計上のサブスクがあるか」をチェックし、該当があれば支出レコードとして自動追加する
- 自動計上済みかどうかは各サブスクごとに「最終計上月」のフラグで管理する
- 自動計上された支出レコードはその後、通常の支出と同じように**編集・削除が可能**
- 月末日の扱い: 計上日として31日が指定されている場合、31日が存在しない月は**その月の最終日に繰り上げ**て計上する

#### 3.1.3 変動固定費の登録・入力促し

- ユーザは「変動固定費マスター」に名前・毎月の入力促し日を登録する(金額は登録しない)
- 指定日になったらホーム画面にバナーを表示し、プッシュ通知でも入力を促す
- **入力されるまでバナーは表示され続ける**(翌月の指定日を過ぎても未入力なら複数のバナーが並ぶ)
- ユーザが金額を入力した時点で通常の支出レコードとして確定する

### 3.2 収入管理

- 入力項目: 日付・タイトル・金額・カテゴリ・メモ
- カテゴリのプリセット: 給与・ボーナス・その他
- カテゴリはユーザが追加可能
- 収入に支払い方法(入金先)の分類は**持たない**

### 3.3 表示・集計

#### 3.3.1 ホーム画面のタブ構成

| タブ | 表示内容 |
|------|----------|
| 日別(デフォルト) | 日別のリスト表示 / カレンダー表示の切り替え |
| カテゴリ別 | 上半分に円グラフ、下半分にカテゴリ別リスト(当月のみ) |
| 月間 | 月の合計支出、1日平均、月ごとの推移グラフ |

#### 3.3.2 家計簿一覧画面

ホームとは役割を分け、**検索・フィルタ専用の画面**として機能する。

想定フィルタ条件:
- 期間(年月範囲)
- カテゴリ
- 金額範囲
- キーワード(タイトル・メモの部分一致)
- 収入 / 支出の別
- 支払い方法

### 3.4 設定

- カテゴリの追加・編集・削除
- 支払い方法の追加・編集・削除
- サブスクの管理
- 変動固定費の管理
- データのエクスポート(CSV / JSON)

### 3.5 補助機能

**簡易電卓**: 支出登録時に金額欄から電卓モーダルを呼び出し、四則演算の結果を金額に反映できる。

## 5. データ概念モデル

詳細なプロパティやリレーションは詳細設計で決定するが、基本設計段階で以下のエンティティが必要であることを明確にしておく。

| エンティティ | 役割 |
|-------------|------|
| 支出レコード | 個々の支出(都度・サブスク自動計上・変動固定費の確定後を含む) |
| 収入レコード | 個々の収入 |
| カテゴリ | 支出・収入それぞれのカテゴリ(プリセット + ユーザ追加) |
| 支払い方法 | 現金、PayPay、クレジットカードなど(プリセット + ユーザ追加) |
| サブスクマスター | 定期的に自動計上するサブスクの定義 |
| 変動固定費マスター | 指定日に入力促しを出す定期費用の定義 |

### 5.1 プリセット

**支出カテゴリ**: 食費 / 日用品 / 交通費 / 娯楽 / サブスク / その他

**収入カテゴリ**: 給与 / ボーナス / その他

**支払い方法**: 現金 / PayPay / クレジットカード

## 6. 主要な業務ルール

### 6.1 月の締め

月初〜月末(1日〜末日)で固定。給料日締めなどのカスタマイズには対応しない。

### 6.2 クレジットカードの計上タイミング

クレジットカードで支払った支出は、**実際に使った日(利用日)に計上する**。引き落とし日への繰り延べは行わない。

**理由**: コンセプトである「実感を残す」に基づき、「いつ何に使ったか」を優先するため。月の支出合計は実際のキャッシュフローとは一時的にズレるが、これは仕様とする。

### 6.3 自動計上・通知の発火タイミング

- サブスク自動計上: アプリ起動時にチェック
- 変動固定費バナー: アプリ起動時にチェックして表示
- 変動固定費のプッシュ通知: 指定日の定刻に配信(時刻は詳細設計で決定)

### 6.4 カテゴリ・支払い方法の削除

ユーザが追加したカテゴリ・支払い方法の削除ポリシーは**未決事項**として詳細設計に持ち越す(既存レコードで使用中のものをどう扱うか)。

## 7. 非機能要件

### 7.1 データ保護

金銭データを扱うため、データの喪失リスクを最小化する。

- **プライマリ保存**: 端末ローカル + iCloud同期(Appleの仕組みを活用しコストゼロでバックアップ)
- **セカンダリバックアップ**: 設定画面から手動でCSV / JSONエクスポートできる機能を用意

### 7.2 オフライン動作

外部APIに依存しないため、すべての機能はオフラインで完結する。

### 7.3 プライバシー

口座連携やサーバ送信を行わないため、金銭データは端末およびユーザのiCloudアカウント内にのみ存在する。

## 8. スコープ外(将来拡張候補)

以下は基本設計の対象外とするが、将来的な拡張可能性を考慮してデータモデル設計に影響する場合は、詳細設計で拡張ポイントを残しておく。

### 8.1 AI予算アドバイス機能

月末の予測や気づきの提示など、LLMを活用した予算管理機能。単純な予算設定はコンセプト上の効果が薄いため、AI連携とセットで将来検討する。

### 8.2 グループ共有機能(旅行モード)

- 方式: リンク共有による非リアルタイム編集型(Walica相当)
- 用途: 旅行など一時的な複数人での立て替え管理
- 実装時の配慮: 詳細設計の段階で、支出レコードに「グループ識別子」のような拡張フィールドを追加できる構造にしておく

## 10. 基本設計の合意事項まとめ

- コンセプトは「実感を残す家計簿」。口座連携はしない
- 支出は「都度 / サブスク / 変動固定費」の3タイプで扱う
- カテゴリ・支払い方法はプリセット + ユーザ追加方式
- ホームは4タブ(日別・カテゴリ別・月間・※円グラフはカテゴリ別に統合)
- 家計簿一覧は検索・フィルタ専用画面として独立
- クレジットカードは利用日で計上する
- データ保護はiCloud同期 + 手動エクスポートの二重体制
- iPhoneをメインターゲットとし、タブバー形式のナビゲーション
- AI予算機能・グループ機能は将来拡張として今回は実装しない
